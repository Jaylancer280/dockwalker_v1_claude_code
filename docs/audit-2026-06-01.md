## CRITICAL

### S1 — Unauthenticated requests can impersonate any user via spoofed identity headers

> **STATUS: ✅ FIXED (2026-06-01).** The strip now runs for every `/api` request via `buildSanitizedApiHeaders` (`lib/auth/api-identity-headers.ts`), called from `middleware.ts`. Unit tests assert forged headers are dropped when unauthenticated and verified identity is applied when authenticated. Full suite green (1560 tests).

- **File:** `apps/web/src/lib/supabase/middleware.ts:33–74` (+ `lib/auth/require-domain-user.ts:44–66`, `lib/auth/require-auth-session.ts:29–35`)
- **Dimension:** security
- **What:** The middleware strips attacker-supplied `x-user-id` / `x-person-id` / `x-current-hat` / `x-identity-type` / `x-blocked` headers and replaces them with verified values — but the entire strip-and-set block is inside `if (user)`. When the caller has **no session** (`user === null`), the block is skipped and the middleware returns `NextResponse.next({ request })` built from the **original inbound headers**, forwarding the attacker's `x-*` headers untouched to the route handler.
- **Why it's exploitable:** Both auth guards trust those headers on the fast path **without calling `getUser()`**. `requireDomainUser` returns `ok` when all four identity headers are present; `requireAuthSession` returns `ok` with only `x-user-id`. The existence of the explicit `requestHeaders.delete('x-user-id')` strip code confirms the team knows these headers otherwise reach the handler.
- **Impact:** An unauthenticated attacker sets `x-user-id: <victim-uuid>` (plus the three person headers) and acts as that victim on any non-admin route whose mutation is keyed on the guard identity. Verified concrete case: **`POST /api/experiences`** writes `EXPERIENCE.ADDED` via the RLS-bypassing `serviceClient` keyed on the spoofed `user.id` — an attacker injects experiences into any victim's profile with no session. Victim UUIDs are readily available (applicant lists, profile/CV responses expose `person_id`).
- **Not affected:** `requireAdmin` re-queries `persons.is_admin` through the **cookie-bound RLS client**, which has no `auth.uid()` for a sessionless attacker, so admin routes fail closed. Routes that gate on an RLS-client read of the resource (e.g. `POST /api/messages/[engagementId]`, protected by the `active_engagements` RLS policy) are incidentally safe — but every route that writes via `serviceClient` keyed on guard identity without a secondary RLS-gated ownership read is exposed.
- **Fix:** Strip the five `x-*` headers **unconditionally** for every `/api/` request before any handler runs — move the `delete(...)` calls outside `if (user)` and always rebuild the forwarded request with the sanitized set (re-adding verified identity only when `user` is non-null). Defense-in-depth: have the guards fall back to `getUser()` when the cookie session is absent, or gate the header fast-path on an unforgeable middleware-only nonce.

## HIGH

### S2 — NDA vessel IMO leaks to shortlisted (not-yet-selected) crew via permanent shortlist chat

> **STATUS: ☑️ ACCEPTED — BY DESIGN (2026-06-01).** Product owner confirmed shortlisted candidates _should_ see NDA vessel details in the pre-selection chat; the current behaviour already does this (`permanent-summary-card.tsx:59-61` renders the IMO; no client mask). The technical write-up below is retained for the record but is **not a defect**. **One follow-up recommended:** no UI warns the employer that opening a shortlist chat reveals the vessel's identity to the candidate (`permanent/[id]/review/page.tsx:614-628` fires "Open chat" with no notice). `CLAUDE.md`'s "revealed only after selection" wording needs human reconciliation.

- **File:** `apps/web/src/app/api/messages/[engagementId]/context/route.ts:19–44, 209–228` + `supabase/migrations/00074_vessels_rls_read_access.sql:23–47`
- **Dimension:** security · **Invariant broken:** "NDA IMO revealed to crew only after **selection** (permanent)."
- **What:** `PERMANENT.SHORTLIST_CHAT_OPENED` is a deliberate **pre-selection** vetting chat (only allowed for `status='shortlisted'` applications). Its projection (00136:713–728) inserts the engagement row with `status='active'`, `phase='shortlist'`, and the shortlisted crew as `crew_person_id`. The context route embeds `permanent_postings → vessels(... imo_number ...)` via the RLS client and returns it **raw** (`...engagement` spread) with no NDA mask on the daywork/permanent branch (only the separate `reference_contact` branch masks). The vessels RLS policy "Engaged users can read engagement vessels" gates on `ae.status='active'` **only — it does not check `ae.phase`**.
- **Impact:** A merely-shortlisted crew member calls `GET /api/messages/{shortlistEngagementId}/context`, passes the membership check, and because their shortlist row is `status='active'`, RLS unmasks the NDA vessel — they read the **real IMO** before the employer has selected them.
- **Fix (two layers):** (1) DB — tighten the 00074 policy to also require `ae.phase = 'active'` (exclude `phase='shortlist'`). (2) Code — in the context route's permanent/daywork branch, route NDA vessel fields through `get_vessel_public` / mask `imo_number` when `phase='shortlist'`. Add a regression test: shortlisted-not-selected crew must receive `imo_number = null` for an NDA vessel. **Note:** `get_vessel_public` itself (00132:109–115) gates permanent reveal on `status in ('active','completed','closed')` with no phase filter, so it needs the same phase fix — the leak lives in the shared gate, not just this route.

### S3 — `POST /api/experiences` trusts caller-supplied `vesselId` → NDA IMO leak chain

> **STATUS: ✋ WON'T FIX — ACCEPTED RISK (2026-06-01).** Product owner's call: NDA is only light protection against casual applicant lookups, and a vessel's IMO is independently discoverable via public registries / IMO search engines, so this channel exposes nothing genuinely secret. Additionally, the legitimate flow lets crew attach **any admin-confirmed vessel** found via IMO search (not just owned ones — `vessels/lookup` → `get_vessels_public_batch`), so the originally-proposed `owner_person_id === user.id` check would break that flow. Left as-is by decision. Technical chain retained below for the record.

- **File:** `apps/web/src/app/api/experiences/route.ts:187–192, 303–304`
- **Dimension:** security (filed low; **verifier upgraded to high** after tracing the full chain)
- **What:** The handler validates that `vesselId` is **present** but never checks the vessel exists or that `owner_person_id === user.id`, then writes it straight into the `EXPERIENCE.ADDED` payload. The DB FK only blocks fabricated UUIDs — a real vessel UUID owned by someone else (including NDA-flagged) passes.
- **Exploit chain (verified end-to-end):** `daywork/discover` returns the raw `vessel_id` for every NDA daywork (only name/IMO are masked). Crew grabs an NDA vessel's UUID → `POST /api/experiences` with it (no ownership check, FK passes) → a `crew_experiences` row now references that NDA vessel → vessels RLS policy #3 ("Crew can read their experience vessels", 00074:49–57) grants the attacker SELECT on that NDA vessel → `GET /api/experiences` joins `vessels(... imo_number ...)` and returns the IMO raw to the owner (no server-side NDA mask on this route). Defeats the NDA invariant with no acceptance/selection.
- **Fix:** Before appending `EXPERIENCE.ADDED`, fetch the vessel with `.eq('id', vesselId).eq('owner_person_id', user.id)` (or the documented shared-vessel rule) and reject 400/403 otherwise — mirroring the lookup already used in `onboarding/route.ts:257–262` and `vessels/[id]` PATCH.

## MEDIUM

### S4 — Open redirect in OAuth/email callback (`next` param unvalidated)

> **STATUS: ✅ FIXED (2026-06-01).** `next` is now validated through `safeRedirectPath()` in `auth/callback/route.ts`; open-redirect regression tests added (`@evil.com` / `//evil.com` → same-origin fallback; legit relative path preserved).

- **File:** `apps/web/src/app/auth/callback/route.ts:9, 46–52, 85, 97`
- **What:** `next` is read straight from the query string and concatenated as `${origin}${next}` with no validation, while the codebase already ships `safeRedirectPath()` (used by onboarding + middleware). Because `origin` has no trailing slash, `?next=@evil.com` produces `https://app.dockwalker.io@evil.com` — a valid URL whose **host is `evil.com`** (the app domain becomes userinfo). The just-authenticated user is redirected to the attacker site (phishing handoff with a valid session). Verified empirically via the WHATWG URL parser. (The `\evil.com` backslash variant does **not** work under WHATWG normalization — only the `@` vector.)
- **Fix:** `const next = safeRedirectPath(searchParams.get('next')) ?? '/onboarding';` — the helper already rejects values not starting with a single `/` and blocks `//`, `..`, `:`.

### S5 — `get_unread_counts(p_person_id)` SECURITY DEFINER RPC is callable for any user

> **STATUS: ✅ FIXED (2026-06-01) — migration 00141.** Function now derives identity from `auth.uid()` (the `p_person_id` arg is retained only for call-signature compatibility and ignored); EXECUTE revoked from `public`/`anon`, granted to `authenticated`/`service_role`. Proven no-regression: all three callers use the user-bound anon client and pass their own `user.id`, so results are byte-identical. Reversible rollback restores the original body + PUBLIC grant.

- **File:** `supabase/migrations/00058_unread_counts_function.sql:9–30`
- **What:** `SECURITY DEFINER` (bypasses RLS), filters on the **caller-supplied** `p_person_id` rather than `auth.uid()`, and has **no `REVOKE`** in the migration → defaults to PUBLIC EXECUTE and is exposed via PostgREST. App callers always pass `user.id`, but any authenticated user can `POST /rest/v1/rpc/get_unread_counts` with a victim's UUID and enumerate that victim's per-engagement unread counts (which engagements they're in + how many unread). Metadata only, no message content (hence medium).
- **Fix:** Drop the param and use `auth.uid()` internally (or guard `if p_person_id <> auth.uid() then raise exception`), then `revoke execute ... from public, anon` / `grant ... to authenticated`. The project already applies this pattern to `custom_access_token_hook` (00078:74) and `admin_delete_person` (00095:86–87). Add a rollback file.

### S6 — WhatsApp OTP: weak code generation + no verify throttling (phone-channel verification bypass)

- **Files:** `apps/web/src/app/api/notifications/whatsapp/register/route.ts:46–48`, `.../verify/route.ts:22–45` _(independently found by 3 finders)_
- **What:** The OTP is generated with non-cryptographic `Math.floor(100000 + Math.random()*900000)`, and the verify route compares it with **no per-attempt counter, no lockout, and no rate limiter** (only `register` is throttled, 3/hour). The code stays valid for the full 10-minute window and is **not** cleared/rotated on a wrong guess, so the 10⁶ space is brute-forceable. `register` stores any caller-supplied E.164 number with no proof of ownership.
- **Impact (scoped accurately by the verifier):** The channel is keyed to the **attacker's own** `person_id`, so a successful brute-force routes the _attacker's_ notifications to a number they don't own — an **abuse/spam vector** through the platform's trusted WhatsApp sender, not account takeover. Real but bounded.
- **Fix:** Generate with `crypto.randomInt(100000, 1000000)` (codebase already uses `crypto.randomBytes` for Telegram tokens — this is an inconsistency); add a `verification_attempts` counter to `notification_channels` with lockout after ~5 failures and code invalidation; apply the existing `lib/rate-limit.ts` helper to the verify route; use constant-time comparison.

### Q1 — Resource-ownership / engagement-membership checks copy-pasted across ~18 routes

- **File:** representative `apps/web/src/app/api/messages/[engagementId]/route.ts:33,81` (+ 8 message routes, 10 `daywork/[id]/*` routes)
- **Dimension:** quality (maintainability / authz-drift risk)
- **What:** The membership check `engagement.crew_person_id !== user.id && engagement.employer_person_id !== user.id → 403` is duplicated verbatim across 8 message routes; the posting-ownership check `daywork.poster_person_id !== user.id` across 10 daywork routes. No shared, tested choke-point exists. **This matters more than typical duplication because the daywork mutation routes append events via the RLS-bypassing `serviceClient`** — the in-handler check _is_ the authorization gate, not defense-in-depth behind RLS. One copy drifting (e.g. a future edit fetching with the wrong client or dropping a status clause) opens a hole the other 17 copies mask in review.
- **Fix:** Extract `requireEngagementMember(engagementId, user, { client, requireStatus? })` and `requireDayworkOwner(dayworkId, user, supabase)` helpers (alongside `require-domain-user.ts`) returning the row or a typed 403/404, and migrate the routes onto them.

## LOW

### S7 — `increment_advisor_usage(p_person_id, ...)` SECURITY DEFINER trusts caller-supplied id

`supabase/migrations/00061_atomic_advisor_usage.sql:8–31`. Same class as S5 (definer, no `REVOKE`, identity from argument). An attacker can inflate a victim's monthly advisor `question_count` to exhaust their free-tier quota (write-side griefing, ~10 calls, resets monthly). **Fix:** same as S5 — `auth.uid()` internally + `revoke execute from public`.

### S8 — PostgREST filter injection via admin vessel `search` param

`apps/web/src/app/api/admin/vessels/route.ts:10,27–29`. Raw `search` is interpolated into a `.or('name.ilike.%${search}%,imo_number.ilike.%${search}%')` string; a comma/filter-syntax payload injects extra OR conditions. **Admin-gated and read-only**, so no privilege escalation — but a real injection sink. **Fix:** strip/reject PostgREST metacharacters (`, ( ) * . :`) before interpolation, or use a single value-positioned `.ilike()` like the canonical route.

### S9 — `locations/canonicalize` lets any authenticated user write global canonical data

`apps/web/src/app/api/locations/canonicalize/route.ts:77–200`. Gated only by `requireAuthSession` (any logged-in user, **including pre-onboarding**); writes attacker-controlled `name` into the shared `regions`/`cities` tables via `serviceClient` (RLS bypass). Format-validated but not content-validated → global location-data pollution / offensive-name injection visible in every user's search. **Fix:** require `requireDomainUser`, rate-limit per user, queue OSM-fallback rows into the existing pending-locations moderation table, and sanitize the display name.

### S10 — Admin canonical insert/update mass-assignment

`apps/web/src/app/api/admin/canonical/[table]/route.ts:121–123,164–176`. POST/PATCH pass the **entire** client body into a `serviceClient` insert/update on the (allowlisted) canonical tables with no column whitelist — any column settable, and typo'd fields surface as raw Postgres 500s. Admin-bounded. **Fix:** per-table writable-column allowlist; pick only those keys; reject unknown fields with 400.

### S11 — `daywork/[id]/invite` doesn't validate target is crew

`apps/web/src/app/api/daywork/[id]/invite/route.ts:49–58`. Checks only that a `profiles` row **exists**, not `identity_type='crew'` — so an agent (who also has a profile row) can be invited. The sibling QR-hire path (`daywork/route.ts:251–267`) checks `identity_type` + deactivated/blocked. Bounded: an agent can't accept (respond route requires `current_hat='crew'`), so worst case is an unwanted invite consuming a slot. **Fix:** mirror the sibling — query `persons` for `identity_type='crew'` and null `deactivated_at`/`blocked_at`.

### S12 — `references/[id]/decline` weaker authorization than `accept`

`apps/web/src/app/api/references/[id]/decline/route.ts:12–46`. Loads the reference via `serviceClient` and checks only `status='pending'` — no referee/email-match check that `accept`, `revoke`, `comment`, and `reference-contacts/decline` all enforce. The projection branch (00136:1073–1076) is equally unguarded vs. the gated `REVOKED_BY_REQUESTER` branch. Realistic exploit requires possessing the opaque share-link **token** (not an enumerable id), so a wrong/forwarded-link holder can kill a pending reference (feature DoS). **Fix:** mirror `accept` — require email-match (or referee/token-holder) in both the route and the projection branch.

### S13 — `references/by-token` returns NDA `snapshot_vessel_imo` to unauthenticated link-holder

`apps/web/src/app/api/references/by-token/[token]/route.ts:20–82`. This public, no-auth, service-role endpoint returns the snapshotted vessel IMO+name pre-consent for the `/ref/[token]` landing page. Migration 00130 deliberately allows references against NDA vessels and snapshots the real IMO. **Partly by design** (00130 frames referee + requester as "insiders" and the preview as a "party-only surface"), and the token is an unguessable 122-bit UUID — but a forwarded/leaked link exposes one NDA vessel's IMO+name to a non-referee before identity verification. **Fix (hardening):** mask `snapshot_vessel_imo` for NDA vessels until the viewer's session email matches `claimed_referee_email` (reuse the page's existing `emailMismatch` logic), or reveal only after Accept. If kept as-is, document the explicit threat-model decision.

### Q2 — `admin_delete_person()` + scrub helpers `DELETE FROM events` (append-only violation; dead-on-arrival)

`supabase/migrations/00095_admin_delete_person.sql:80–82` (+ same pattern in `00099:179`, `00106:115`, `00107:149`). These `SECURITY DEFINER` utilities attempt `delete from public.events`, which the unconditional `events_no_delete` BEFORE-DELETE trigger (`00001:31–33`) rejects for **all** roles — so the function raises and rolls back for any user who has events (i.e. everyone real). It's dormant (revoked, not wired to any route; the live admin DELETE correctly uses `PERSON.DATA_SCRUBBED`), but it's documented as the manual GDPR/test-cleanup tool and would fail confusingly from psql. (Independently corroborated by `scripts/stress-test-cv-builder.ts`, which calls it a "dead-end RPC".) **Fix:** remove the `delete from public.events` lines (keep the ledger, as the live path does), or — if true erasure is required — wrap in an audited `set local session_replication_role = replica;` and document it as the single sanctioned exception. Don't ship a function that can never succeed.

### Q3 — Admin audit-log event writes are fire-and-forget with errors swallowed

`apps/web/src/app/api/admin/canonical/[table]/route.ts:130–137,183–190`. `appendEvent(...).catch(() => {})` — unawaited and errors fully discarded. If it throws, the canonical row still mutates but no `ADMIN.CANONICAL_ADDED/UPDATED` audit row is written and nothing reaches Sentry; the unawaited promise can also be dropped on serverless teardown even on success. **Fix:** `await` the call inside the try (or `waitUntil`), and replace the empty catch with `Sentry.captureException`.

### P1 — `daywork/discover` builds an unbounded `NOT IN (excludedIds)` filter

`apps/web/src/app/api/daywork/discover/route.ts:69–96`. Fetches **every** application row for the crew member across 6 statuses (incl. `viewed` — added by merely browsing a card, never pruned) with **no LIMIT**, then serializes the full UUID list into a `id NOT IN (...)` clause on **every** discover request (including each infinite-scroll batch). The list grows monotonically over an active account's lifetime, enlarging the request, plan, and negative scan on the hottest crew read path. **Fix:** replace the client-side anti-join with a server-side `NOT EXISTS` / `LEFT JOIN ... IS NULL` in an RPC/view, or a materialized `seen` projection keyed on `(crew_person_id, daywork_id)` joined at the DB layer.

## <a name="verified-clean-areas"></a>Verified-clean areas (high confidence)

These were specifically examined and found sound — useful to record so they aren't re-litigated:

- **Webhooks** — `webhooks/stripe` verifies the Stripe signature with `constructEvent(body, sig, secret)` before trusting the body and resolves `person_id` from signed metadata + customer lookup (never the raw body); `webhooks/telegram` verifies `x-telegram-bot-api-secret-token` and gates linking on single-use, expiring, person-scoped tokens.
- **Cron** — all 6 routes (`data-scrub`, `document-cleanup`, `availability-expiry`, `engagement-starts`, `reference-expiry`, `invitation-expiry`) require `Authorization: Bearer ${CRON_SECRET}` and 401 when unset/mismatched. No unauthenticated path to the GDPR scrub or mail-send jobs.
- **Admin gating** — every `admin/**` route calls `requireAdmin` (re-queries `persons.is_admin` via the RLS client) on **every** exported method; multi-method files individually verified. `admin/canonical/[table]` validates the `:table` param against a fixed allowlist before `serviceClient.from(table)`. `notes/[noteId]` PATCH enforces both note→person ownership and authoring-admin.
- **Billing** — `create-checkout` / `create-portal` / `status` scope every query and Stripe customer to the authenticated `user.id`; no body-supplied `person_id`.
- **Notification channels, support, reports, account export/deactivate, push-tokens, notifications/read\*** — all reads/writes scoped to `user.id` via the RLS client with UUID validation and batch caps; no IDOR.

## False positives (raised then rejected on verification)

- **`profile/[personId]` "leaks NDA experience vessel name+LOA"** — **rejected.** The query uses the **RLS-scoped** client; vessels RLS strips NDA rows for a relationship-only viewer (policies grant NDA access only to the owner, the crew themselves, or an active-engagement party), so the viewer receives `null`. `imo_number` is never selected. (Contrast the public CV endpoint, which uses `serviceClient` and therefore _must_ — and does — mask in JS.)
- **`messages/.../context` "NDA mask depends on one RLS clause, fix by using `get_vessel_public`"** — **rejected as filed.** The proposed RPC has the _same_ phase-blind gate, so it wouldn't fix anything, and the "currently safe" premise was wrong (the real leak is S2 above). The genuine issue is captured correctly as **S2**.

The volume is driven by genuine feature surface, not bad writing; the cleanup upside is modest (~single-digit %) and is best pursued for the _drift-safety_ of consolidating the authz checks rather than for line-count reduction alone.
